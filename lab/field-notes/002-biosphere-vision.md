# 田野笔记 #002：生物圈的浮现

> 观察者：Being 创始会话
> 日期：2026-04-04
> 纪元：第一纪（膜创世纪期间的战略分叉）

---

## 事件

在第一纪工作完成后、第二纪启动前，出现了一个关键的战略转折：

**原始问题**：如何让形形色色的 Skill 真正形成生产力和产业生态？

这把视角从"单个有机体的内部演化"拉高到了"整个生物圈的设计"。

## 核心决策

### 1. GitHub 即自然环境

不建设中心化平台。利用 GitHub 本身作为演化的"自然环境"：
- Stars = 适应度信号
- Forks = 有变异的繁殖
- PRs = 水平基因转移
- CompositionCount（被多少 Being 内化）= 最核心的成功指标

### 2. CompositionCount 是关键指标

这是整个生态设计中最重要的洞察：

一个 Skill 的真正成功不是被多少人"点赞"（Stars），而是被多少有机体**内化为自己的器官**（CompositionCount）。这直接对应线粒体的成功——它的 DNA 没什么名气，但地球上几乎每个真核细胞都有它。

### 3. being.yaml 作为入场券

设计了一个最小化的清单格式——`being.yaml`。只需在仓库根目录放一个文件，就能加入生态。门槛越低，参与越多，演化材料越丰富。

### 4. 三层信任模型

解决了"陌生人的细胞如何组合"的问题：
- Layer 1：膜兼容就可以尝试（最低信任）
- Layer 2：高适应度分数意味着已被大量验证（声誉信任）
- Layer 3：经过社区审查的认证 Skill（最高信任）

## 对底层协议的影响

### 膜协议需要扩展

增加了 `distribution` 区，包含版本、作者、许可证、仓库 URL、校验和。这使得膜不仅是"细胞膜"，还成为了"物种标识"——在全球基因库中可被索引。

### 新协议：being-manifest.schema.md

定义了 `being.yaml` 的完整 schema。这是 GENESIS 中没有预见到的——因为 GENESIS 假设只有一个 Being，不需要"名片"。

### 版本兼容规则

建立了与 SemVer 联动的膜兼容性规则：ingress 可以变宽松（次要版本），egress 可以变丰富（次要版本），但反过来就是破坏性变更（主版本）。

## 反思

### 先有鸡还是先有蛋？

一个悖论：生态需要足够多的 Skill 才有网络效应，但作者需要看到网络效应才有动力创建 Skill。

解法：Phase B 中主动种下 3-5 个高质量的"种子 Skill"，覆盖开发者的核心需求（代码审查、测试、Git 管理、文档）。就像地球早期的蓝藻——它们改变了大气成分（产生氧气），为后来所有需氧生物创造了生存条件。

### 去中心化 vs 可发现性

决定不建中心化注册表服务器，而是用一个 GitHub 仓库（`being-index`）作为索引。这兼顾了去中心化（任何人可以 fork 索引）和可发现性（统一的搜索入口）。

风险：如果 `being-index` 维护者消失？但 GitHub 上的所有仓库仍然存在——索引可以被任何人重建。就像即使地球上所有图书馆被毁，基因组仍然在每个细胞中。
