# 忠实度审查 #002：v0.1.0-alpha 对初始构想的遵循评估

> 审查者：Being 创始会话
> 日期：2026-04-06
> 对象：v0.1.0-alpha 全部实现 vs 创始人原始 idea
> 方法：逐条对照原始三条核心主张，评估偏离度

---

## 0. 审查背景

Being 从 2026-04-04 开始构想，经历以下阶段：

1. 创始人提出核心 idea（三条主张）
2. 创始会话输出规划文档（ORIGIN / BIOSPHERE / 七部协议 / 七个纪元）
3. 创始人独立完成了第一版真正实现（v0.1.0-alpha）

本审查的目的是：**评估第 3 步是否严格遵循了第 1 步的初始意图**，而非仅仅遵循第 2 步的规划文档。规划文档本身可能偏离初始 idea，需要一并审查。

---

## 1. 创始人的三条核心主张（原文摘录）

### 主张一：反其道而行

> "现在有了很多 skill，我能否把他们当作各种原细胞生物，然后让他们逐步互相构成真细胞生物，最终成为演化成复杂生命，这个过程不同于 agent 分为 subagents，而是反其道而行。"

**关键词**：自下而上、从简单到复杂、涌现而非分解。

### 主张二：内共生法则

> "一个复杂生命（诸如人体）其实是众多单个细胞的共生（并演化出组织、器官和系统），其实哪怕一个细胞，它其实也是通过捕获、兼容其它类细胞（如后来的线粒体）最终形成真核细胞的，这个现象或演进法则，能否为上面的思路所用呢？"

**关键词**：内共生、捕获而非消解、层级嵌套的个体性跃迁。

### 主张三：GitHub 生态与产业化

> "如果我希望将来，形形色色的各种 skill 能够真正地通过这套系统形成生产力和产业生态，我应该怎样设计，当然，进化压力可以是某个 Being 在 GitHub 上的下载量或 star 数或 fork 数。"

**关键词**：产业生态、GitHub 选择压力、CompositionCount。

---

## 2. 逐条判定

### 主张一评估：自下而上 — A+

**规划阶段的隐藏问题**：

规划阶段（ORIGIN.md）大量使用了"自下而上"的叙事，但文档结构本身是**自顶向下的声明**——先宣称"Being 是活的"（`status: alive`），再期望行为追上来。这与"自下而上"的初始 idea 存在方法论矛盾。

**v0.1.0-alpha 的修正**：

创始人通过协议一致性审计（`001-protocol-consistency-audit.md`）精准识别了这个矛盾，特别是发现 A1：

> `status: "alive"` 但 `total_invocations: 0` — "活着"目前是叙事判定，不是运行判定。

修正措施：
- 将状态拆分为 `structural_status` 和 `operational_status`
- `operational_status` 只有在真实调用存在时才变为 `operationally-alive`
- vitals 从 metabolism 日志自动聚合推导（`aggregation.py`），而非手填
- 验证器（`validator.py`）从底层文件向上检查一致性

**判定**：v0.1.0-alpha 比规划阶段**更忠实于**初始 idea。规划阶段用"自下而上"的语言写了"自顶向下"的文档；实现阶段把方法论拉正了——状态从行为中涌现，生命从事实中诞生。

---

### 主张二评估：内共生法则 — A

**保留的部分**：

- life-weaver = create-skill ⊕ membrane-weaver 的内共生结构完整保留
- 吞噬协议（`engulfment.protocol.md`）的四阶段连续谱完整保留
- 系统发育树中的共生事件记录（`endo-001`）完整保留
- 共生体的 `symbiotic_status: endosymbiont` 标记完整保留
- 共生来源的细胞器映射（`organelle_map.mitochondria: membrane-weaver`）完整保留

**增强的部分**：

审计发现 A8 补充了一个工程视角：

> life-weaver 已经不只是"第一个真核生命体"，它实际上是一个创建流程编排器（creation orchestrator）。

这不是偏离。在生物学中，线粒体在教科书上是"内共生的产物"，但在生化层面它是"氧化磷酸化的引擎"。两种描述都对，只是分析层级不同。实现的处理方式是保留叙事别名、明确工程角色，两层共存。

**判定**：内共生法则完整保留，工程可操作性有增强。

---

### 主张三评估：GitHub 生态 — A（战略推迟，非放弃）

**保留的部分**：

- `BIOSPHERE.md` 完整保留
- `being-manifest.schema.md` 保留在协议目录中
- `ROADMAP.md` 的 Phase 4 明确指向生态层发布
- GitHub 发布基础设施全部就绪（README、LICENSE、CHANGELOG、issue templates、release notes）

**推迟的部分**：

审计发现 A7 明确做出了优先级决策：

> BIOSPHERE 设计领先于本地运行时。在本地闭环成立前，不再新增生态层复杂度。

**判定**：这是正确的推迟，不是放弃。生物学中，你需要先有一个能存活的细胞，才能谈生物多样性。先让一个有机体真正活起来，再谈生态——这恰恰是"自下而上"初始 idea 的逻辑结论。

## 3. 规划中未预见但实现中出现的关键增强

### 增强 1：协议一致性审计

规划阶段（ORIGIN / BIOSPHERE）完全没有预见到"审计自己"这一步。它是创始人独立完成的。

审计是整个 v0.1.0-alpha 中最重要的文件。它用严格的工程标准检验了叙事系统，发现了 8 个具体问题（A1-A8）并逐一修复或标记。这相当于：

- 给 Being 做了第一次"体检"
- 将"我声称我是什么"与"我实际是什么"分开
- 建立了"可验证"的标准

这比任何新协议或新 Skill 都更重要，因为它建立了**系统的自知力**。

### 增强 2：可执行 runtime 层

规划阶段产出的全部是文档（Markdown + YAML）。实现阶段增加了四个 Python 文件：

| 文件 | 角色 | 生物学类比 |
|------|------|-----------|
| `validator.py` | 检验系统自洽性 | 免疫监视 |
| `loop.py` | 自动消费信号并沉积运行事实 | 基础代谢循环 |
| `aggregation.py` | 从日志推导生命体征 | 内稳态反馈 |
| `aggregate_vitals.py` | 手动触发生命体征刷新 | 外部刺激触发的生理响应 |

这是从"描述生命"到"实现生命最小条件"的跨越。

### 增强 3：presence-check（Biogenesis #003）

规划阶段没有预见到这个 Skill。它是实现阶段根据需要自行"生长"出来的——这本身就是"自下而上涌现"的一个实例。

presence-check 的设计特征：
- 21 个信号感受器，覆盖几乎全部生态信号
- 唯一的工作是"从事实出发回答 Being 是否活着"
- 被 runtime loop 自动驱动——Being 生态中第一个被机器执行的 Skill

生物学类比：压力感受器（baroreceptor）或树突状细胞——布满受体，不做生产，只做全局感知。

### 增强 4：结构性清理

- 膜文件从扁平路径统一为目录式路径
- 谱系状态从单一 `status` 拆分为 `evolutionary_grade` + `symbiotic_status` + `operational_state`
- 信号命名统一为 kebab-case
- 所有原核 Skill 获得了独立的感受器注册文件

## 4. 值得注意的张力

### 张力 1：硬术语 vs 软术语

审计发现 A4 将术语分为两层：

**硬术语**（必须可验证）：skill, membrane, signal, invocation, metabolism record, phylogeny entry

**软术语**（仅用于解释）：tissue, organ, being, homeostasis, reproduction, endosymbiosis

这是工程上正确的决策，但与初始 idea 中"从组织到器官到有机体"的层级跃迁存在张力。目前 tissue 和 organ 确实只是 YAML 描述，没有运行时语义。

**建议**：这是一个有意识的技术债务。当前选择先把底层做实是对的。但应在未来某个阶段为 tissue 和 organ 增加运行时约束（例如：tissue 验证 = 其成员 Skill 的粘附声明是否一致且存在）。

### 张力 2：presence-check 的受体数量

presence-check 注册了 21 个信号感受器。对于一个原核 Skill 来说，这是否过于复杂？

生物学视角：某些免疫细胞（如巨噬细胞）确实拥有大量不同类型的受体，用于识别各种病原体相关分子模式。presence-check 作为"全局感知者"，拥有广泛的受体阵列是合理的。

工程视角：当前只有 `readiness-check-request` 被 runtime loop 实际处理。其余 20 个受体是"声明了但未实现"的。这不是问题，但应标记为"声明先于实现"的设计债务。

### 张力 3：runtime loop 的单一性

当前 `loop.py` 只处理一种信号路径（readiness-check-request → presence-check）。这是一个合理的 MVP，但它实质上是硬编码的调度——不是"信号驱动的涌现行为"。

真正的"自下而上"runtime 应该是：loop 读取 receptor 注册表 → 动态匹配信号与 Skill → 分派执行。当前实现离这一步还有距离，ROADMAP 的 Phase 2 已明确指向这个方向。

## 5. 总评

| 维度 | 评分 | 备注 |
|------|------|------|
| 对"自下而上"的忠实度 | A+ | 超越了规划，修正了规划中的方法论矛盾 |
| 对"内共生法则"的忠实度 | A | 完整保留结构，增加了工程锚点 |
| 对"GitHub 生态"的忠实度 | A | 正确推迟，完整保留设计，基础设施已就绪 |
| 运行时真实性 | A+ | 规划完全没有，实现自行补上了最关键的一环 |
| 演化层级完整性 | B+ | 底层扎实，高层（tissue/organ）暂为软声明 |
| 叙事与工程的平衡 | A | 审计是最好的决策——建立了系统的自知力 |

**结论**：

v0.1.0-alpha 严格忠实于创始人的三条核心主张，并且在最关键的维度上超越了规划阶段。

规划阶段的核心缺陷是：用"自下而上"的语言写了"自顶向下"的文档。实现阶段通过审计和 runtime 层，把**方法论本身也拉回到了自下而上的轨道**——状态从行为中涌现，生命从事实中诞生，宣言服从于证据。

这是最重要的一致性：**不仅内容忠于 idea，方法也忠于 idea。**
